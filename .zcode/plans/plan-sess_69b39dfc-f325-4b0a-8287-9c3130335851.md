# Scheduled Crons Feature — Implementation Plan

## Overview

Add **scheduled recurring agent runs** that are managed via the web UI. Each cron job: targets one repo, runs on a user-defined branch with a freeform prompt, and its output is **issues opened by the agent** (e.g. "find bugs and open issues"). This is a new run type alongside the existing issue→PR flow.

User model: **define time (cron expression), define branch, define instructions → get work done.**

---

## Architecture decisions (confirmed)

| Decision | Choice |
|---|---|
| Schedule syntax | **Cron expressions** (`0 0 * * *` = daily midnight) |
| Queue dedupe key | **New `cron_job_id` column** on `jobs` table |
| Engine path | **Separate `runCronJob()`** — leaves the 600-line `runJob` untouched |
| Repo scope | **Per-cron repo** (one repo per cron job) |
| Persistence | **DB-driven** (`cron_jobs` table) — managed via UI, not config file |

## Phase 1 — Data layer

### 1a. New `src/server/cron-store.ts` — `CronStore` class

SQLite table for cron definitions + scheduling state:

```sql
CREATE TABLE IF NOT EXISTS cron_jobs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  repo TEXT NOT NULL,              -- "owner/name"
  prompt TEXT NOT NULL,            -- freeform instructions
  branch_name TEXT NOT NULL,       -- e.g. "wa-agent"
  cron_expression TEXT NOT NULL,   -- "0 0 * * *"
  profile TEXT,                    -- profile name, or NULL for default
  enabled INTEGER NOT NULL DEFAULT 1,
  last_run_at TEXT,
  next_run_at TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Methods (mirroring `RunStore`'s class-over-shared-DB pattern):
- `createCron(input): CronRow`
- `updateCron(id, partial): CronRow`
- `deleteCron(id): void`
- `getCron(id): CronRow`
- `listCrons(): CronRow[]`
- `listDueCrons(now: Date): CronRow[]` — `enabled = 1 AND next_run_at <= now`
- `markScheduled(id, lastRunAt, nextRunAt): void` — after enqueueing
- `recomputeNextRun(id, expr): void` — after create/update/enable

Uses `fromDb()` static factory for in-memory tests (same pattern as `JobQueue`/`RunStore`).

### 1b. Queue migration — `src/server/queue.ts`

Add `cron_job_id INTEGER NOT NULL DEFAULT 0` column via the existing PRAGMA-introspection migration pattern (`migrateAddCronJobId`).

**Rebuild the dedupe index** to include `cron_job_id`:
```sql
-- drop old, create new (in a migration guarded by index introspection)
DROP INDEX IF EXISTS idx_jobs_active_dedup;
CREATE UNIQUE INDEX IF NOT EXISTS idx_jobs_active_dedup
  ON jobs(repo, issue_number, cron_job_id) WHERE status IN ('queued', 'running');
```

Convention:
- **Normal jobs**: `cron_job_id = 0`, `issue_number` = real issue → dedupes on `(repo, issue, 0)`.
- **Cron jobs**: `cron_job_id = N`, `issue_number = 0` → dedupes on `(repo, 0, N)`. Different cron jobs for the same repo get different `cron_job_id`, so they can run concurrently.

New `QueuedJob` field: `cron_job_id: number`. New enqueue method:
```typescript
enqueueCron(opts: { repo: string; cronJobId: number; profile?: string | null; source?: string }): QueuedJob
```
Sets `issue_number = 0`, `cron_job_id = cronJobId`, `source = "cron"`.

### 1c. RunStore migration — `src/server/run-store.ts`

Make `issue` nullable and add cron linkage:
```sql
ALTER TABLE runs ADD COLUMN cron_job_id INTEGER;      -- NULL for normal runs
ALTER TABLE runs ADD COLUMN output_issue_url TEXT;     -- the issue a cron run created
```

`NewRun.issue` becomes `issue?: number`. `RunRow.issue` becomes `number | null`. Add `cron_job_id` + `output_issue_url` to `RunRow` / `RunUpdate`.

## Phase 2 — GitHub client + tools

### 2a. `GitHubClient.createIssue()` — `src/github/client.ts`

```typescript
async createIssue(repo: string, title: string, body: string, labels?: string[]): Promise<{ number: number; html_url: string }>
```
Uses `octokit.rest.issues.create`. Simple addition alongside `createPullRequest`.

### 2b. `createOpenIssueTool()` — `src/engine/tools.ts`

Follows the exact `createCommentOnIssueTool` pattern:

```typescript
export function createOpenIssueTool(gh: GitHubClient, repo: string, cronLabels?: string[]) {
  return defineTool({
    name: "open_issue",
    label: "Open Issue",
    description: "Open a new GitHub issue with a title and body...",
    parameters: Type.Object({
      title: Type.String(),
      body: Type.String(),
      labels: Type.Optional(Type.Array(Type.String())),
    }),
    execute: async (_id, params) => {
      const issue = await gh.createIssue(repo, params.title, params.body, [...(cronLabels ?? []), ...(params.labels ?? [])]);
      return { content: [{ type: "text", text: `Opened issue #${issue.number}: ${issue.html_url}` }], details: {} };
    },
  });
}
```

## Phase 3 — Engine

### 3a. `buildCronPrompt()` — `src/engine/prompt.ts`

New exported function alongside `buildPrompt`:

```typescript
export function buildCronPrompt(prompt: string, repo: string, agentName?: string, sysInfo?: string): string
```

Structure: `sysInfo` block → repo context → load skills (`noodle-default` only; cron runs are investigative, not fix-work) → the freeform prompt as the task → instruction to use the `open_issue` tool to report findings. No `## Issue` / `## Discussion` sections (there's no source issue).

### 3b. `runCronJob()` — new file `src/engine/cron-run.ts`

Separate from `run.ts`. Shares the same building blocks (`Workspace`, `resolveProfile`, `registerCustomProviders`, `SessionManager`, `StallWatcher`, `buildFooter`) but with a different lifecycle:

```typescript
export interface CronRunInput {
  repo: string;
  prompt: string;
  branchName: string;
  profile?: string | null;   // resolved name, or null for default
  jobId?: string;
  token?: string;
}

export async function runCronJob(
  config: NoodleConfig,
  gh: GitHubClient,
  input: CronRunInput,
  deps?: { authStorage?, runStore?, createAgentSessionFn?, tokenProvider?, ghProvider? }
): Promise<RunResult>
```

Flow:
1. Resolve profile (from `input.profile` or `default_profile`).
2. Resolve model via registry (same as `runJob`).
3. Clone repo.
4. **Branch**: create `input.branchName`, or if it already exists on remote, fetch + reset onto its tip (reuse the `checkoutOrReuse` pattern — a daily cron stacks onto yesterday's branch).
5. `installSkills(ws.path)`.
6. Build prompt via `buildCronPrompt`.
7. Create pi session with `customTools: [createOpenIssueTool(gh, repo)]`. **No** `comment_on_issue` (no source issue to comment on).
8. Run with `StallWatcher` (same stall budgets as normal runs).
9. Capture stats + agent answer.
10. **Commit + push** the branch (preserves any exploratory changes for traceability; no PR opened).
11. Record in `RunStore` with `cron_job_id` set, `issue = null`.

Exported helpers (`buildFooter`, `formatDuration`, etc.) are imported from `run.ts` — no duplication.

## Phase 4 — Scheduler

### 4a. Add dependency

`npm install cron-parser` — well-established, ~10KB, no deps. Provides `parseExpression(expr).next()` to compute the next run time from a cron expression.

### 4b. New `src/server/cron-scheduler.ts` — `CronScheduler` class

```typescript
export class CronScheduler {
  constructor(
    private readonly cronStore: CronStore,
    private readonly queue: JobQueue,
    private readonly authProvider: AuthProvider,
  ) {}

  start(): void  // setInterval every 60s → tick()
  stop(): void
  async tick(): Promise<void>  // listDueCrons → enqueueCron each → markScheduled
}
```

`tick()`: for each due cron job, resolve the repo's installation token via `authProvider.forRepo(repo)`, call `queue.enqueueCron({ repo, cronJobId, profile })`, then `cronStore.markScheduled(id, now, nextRunFromCronParser)`. Errors per-cron are caught + logged (one bad cron doesn't stall the others).

Boot: `start()` fires an immediate `tick()` so due crons run on startup, then every 60s. Wired in `serve.ts` alongside the existing issue-scan `Scheduler`.

## Phase 5 — API + serve wiring

### 5a. New API routes in `src/server/ui-routes.ts`

All behind the existing `auth` preHandler:

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/crons` | List all cron jobs (with last_run, next_run, enabled) |
| `POST` | `/api/crons` | Create a cron job (validates cron expression) |
| `GET` | `/api/crons/:id` | Detail: cron def + its recent runs |
| `PATCH` | `/api/crons/:id` | Update (edit fields, enable/disable) |
| `DELETE` | `/api/crons/:id` | Delete |
| `POST` | `/api/crons/:id/run` | Trigger an immediate manual run (enqueue now) |

`UiDeps` gains `cronStore: CronStore`.

### 5b. `serve.ts` changes

- Construct `CronStore` from the shared DB.
- Instantiate `CronScheduler`, call `start()` after workers boot (alongside the existing `scheduler?.start()`).
- **Dispatch in `runJobFn`**: check `job.cron_job_id`. If `> 0`, call `runCronJob` instead of `runJob`. The cron job's `prompt`/`branchName` are looked up from `CronStore.getCron(job.cron_job_id)`.
- Pass `CronStore` through to `registerUiRoutes`.
- Graceful shutdown: `cronScheduler?.stop()`.

## Phase 6 — Web UI

### 6a. New `crons-view` section in `public/index.html`

Added to the `views` array (`["login-view", "runs-view", "chat-view", "crons-view"]`). Two sub-views managed by JS state:

**Cron list** (`#crons-list`):
- Topbar: `🍜 Crons` title, `+` button (add new), `↻` refresh, `⎋` logout.
- Each row: status dot (enabled=green / disabled=grey), name, repo, cron expression (humanized), next run time, last run status. Tap → detail view.

**Cron detail / create form** (`#cron-detail`):
- Fields: name, repo, branch, cron expression, profile (dropdown of configured profiles), prompt (textarea).
- Human-readable schedule preview ("Runs daily at 12:00 AM") computed client-side from the cron expression.
- Buttons: Save, Run Now, Enable/Disable toggle, Delete.
- Below the form: a list of this cron's recent runs (reusing the `.run` row pattern from the runs list), tappable into the chat view.

**Navigation**: Add a nav row or tab bar. The runs-view topbar gets a "Crons" button (or a bottom tab bar with Runs | Crons). Back button from cron detail → cron list.

All styling reuses the existing CSS custom properties (`--bg-elev`, `--border`, `--ok`/`--fail`/`--running`, `--radius`) and class patterns (`.topbar`, `.scroll`, `.dot`, `.badge`, `.run`). The create form reuses `.login-card`-style inputs. No new CSS framework.

### 6b. JS functions

Following the existing `openX()` / `renderX()` convention:
- `openCrons()` → fetch `GET /api/crons`, render list.
- `openCronDetail(id)` → fetch `GET /api/crons/:id`, render form + runs.
- `createCron(formData)` → `POST /api/crons`.
- `saveCron(id, formData)` → `PATCH /api/crons/:id`.
- `deleteCron(id)` → `DELETE /api/crons/:id`.
- `runCronNow(id)` → `POST /api/crons/:id/run`.
- Cron expression humanizer: small inline function or `cronstrue` (can be added as a second tiny dep, or a simple hardcoded mapper for common patterns).
